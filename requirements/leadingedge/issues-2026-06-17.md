## Issue Inventory

### LE-01: Asymmetric Missing-Field Detection

| | |
|---|---|
| **Severity** | High |
| **Frequency** | Observed once (BLL2_MAJ6) |
| **Test doc** | BLL2_MAJ6 |

**Problem:** The comparison only detects fields that are on the **template** but missing
from the **aggregate**. It does not detect fields that are on the **aggregate** but missing
from the **template**.

**Evidence (same analysis, two directions):**

- **Template → Aggregate (detected):** "Surviving Spouse" is listed under Eligibility
  Determination on the template as "Not Covered." It is absent from the aggregate. The
  comparison correctly flags this as "Missing — Missing in Aggregate document."

- **Aggregate → Template (not detected):** The aggregate contains "Elective Abortion,"
  "Therapeutic Abortion," and "Newborn Nursery" — none of which appear on the template.
  These are completely absent from the comparison output. Tia annotated the comparison
  screenshot: "I would expect to see this as missing on the Aggregate" for each line.

**Additional nuance:** The template breaks Abortion into four sub-categories (Inpatient
Institutional, Outpatient Institutional, Outpatient Professional, Office Professional)
while the aggregate uses different categories (Elective Abortion, Therapeutic Abortion).
The comparison only shows the template's breakdown, marking them as "Terminology differs"
mismatches, but never surfaces the aggregate-only categories.

**Expected behavior:** The comparison should be bidirectional — any field present in either
document but absent from the other should be flagged as "Missing."

### LE-02: Non-Deterministic Results Across Runs

| | |
|---|---|
| **Severity** | High |
| **Frequency** | Confirmed on one document pair, two runs |
| **Test doc** | RCP2_HYD1 (same files, same day) |

**Problem:** Running the same two documents through the tool twice on the same day produced
significantly different results.

**Evidence:**

| Metric | Run 1 (morning) | Run 2 (afternoon) | Delta |
|--------|-----------------|-------------------|-------|
| Total rows | 480 | 355 | −125 |
| Match | 275 (57%) | 216 (61%) | −59 |
| Mismatch | 196 | 129 | −67 |
| Uncertain | 1 | 0 | −1 |
| Missing | 8 | 10 | +2 |

The 125-row difference cannot be explained by missing-field count changes alone (+2).
Clayton noted on the call that this suggests the LLM is concatenating row content
inconsistently between runs.

**Additional context (from Joey):** The team was making backend tweaks during this period.
Tia confirmed both runs were on the same day (last week of testing), morning vs afternoon.
This doesn't rule out a prompt or config change between runs, but the documents and upload
process were identical.

**Expected behavior:** The same inputs should produce substantially identical outputs. Minor
variation in phrasing is acceptable; a 26% swing in row count is not.

### LE-03: Benefit Line Concatenation (Insufficient Granularity)

| | |
|---|---|
| **Severity** | High |
| **Frequency** | Observed in BLL2_MAJ6; correct behavior seen in RCP2_HYD1 |
| **Test doc** | BLL2_MAJ6 (problem), RCP2_HYD1 (expected) |

**Problem:** Benefits that should each appear on their own row are concatenated into a
single summary line, preventing line-by-line audit.

**Evidence:**

- **Therapies:** Comparison shows two rows: "Therapies (Cardiac, Chemo Coverage)" and
  "Therapies (Cardiac, Chemo Coverage)" (one for Coverage, one for another setting).
  Expected: ~20 individual rows (Cardiac Rehab Outpatient Institutional, Cardiac Rehab
  Inpatient Professional, Chemotherapy Outpatient Institutional, etc.), each with their
  own INN Deductible / OON Parameters settings.

- **Transplants:** Comparison shows one row: "Transplants (Live Donor, Bone Marrow, Org
  Coverage)." Expected: ~15 individual rows (Bone Marrow Donor Search Fee Non-BDCT,
  Live Donor Health Services Non-BDCT, Organ Transplants Institutional BDCT, etc.).

**Contrast:** The RCP2_HYD1 analysis correctly breaks these out into individual rows,
confirming the model *can* produce the right granularity — it just doesn't do so
consistently.

**Expected behavior:** Every benefit sub-line that appears as a distinct row in the source
documents should appear as its own row in the comparison.

### LE-04: False Match When Qualifier Text Differs in Multi-Column Cells

| | |
|---|---|
| **Severity** | Medium |
| **Frequency** | Observed once (BLL2_MAJ6) |
| **Test doc** | BLL2_MAJ6 |

**Problem:** When the benefit name matches but descriptive/qualifier text in merged columns
A–C differs between the template and aggregate, the comparison still reports "Match."

**Evidence:** Emergency Room entries — the template cell includes the text "Limit combined
Institutional/Professional." in the description area. The aggregate's corresponding cell
does not contain this qualifier. Despite this difference, the comparison marks the row as
"Match."

**Clayton's on-call hypothesis:** The complex formatting in merged multi-line cells
(internal line breaks, mixed formatting) may be obscuring content from the extraction
pipeline. The tool may need to strip and standardize formatting before comparison.

**Expected behavior:** All text content within a cell should be compared, not just the
benefit name. If qualifier text like limits or notes is present in one document but absent
from the other, this should be flagged as a Mismatch (or at minimum, an annotation in the
Details column).

### LE-05: Semantic Equivalence Overriding Literal Comparison

| | |
|---|---|
| **Severity** | Medium |
| **Frequency** | Observed once (RCP2_HYD1) |
| **Test doc** | RCP2_HYD1 |

**Problem:** "12 Months" vs. "365 Days" for Timely Filing is marked as "Match." The AI
treats these as semantically equivalent, which is technically correct in general — but for
FOB audit purposes, the exact language matters.

**Evidence:** Comparison row shows:
- FOB Standard: `12 Months`
- Aggregate Value: `365 Days`
- Status: `Match`
- Details: `12 Months is semantically equivalent`

**Expected behavior:** For this use case, the comparison must be **literal, not semantic**.
If the text strings differ, it should be a Mismatch regardless of whether the values are
logically equivalent. The auditor needs to see every textual discrepancy to flag it to
Anthem.

**Clayton's on-call note:** This is addressable via prompt engineering — providing explicit
positive/negative examples to the model that literal text differences must always be
flagged.

### LE-06: Pre-Certification Lists Systematically Omitted

| | |
|---|---|
| **Severity** | High |
| **Frequency** | Consistent across multiple analyses |
| **Test doc** | RCP2_HYD1 (shown), but reported across most/all test cases |

**Problem:** Pre-Cert Lists are completely absent from comparison output even though they
are present in both source documents.

**Evidence:** Side-by-side screenshot shows:
- Template: "Standard HealthLink Precert Listing 01/01/2023 | 01/01/2023 – 12/31/2023"
  with items including ABA Therapy, Alcohol/Substance Abuse, Ambulance, Diagnostic X-ray
  services, genetic testing, etc.
- Aggregate: "Leading Edge Administrators Standard Precert Listing | 01/01/2026 –
  12/31/9999" with matching item list.
- Comparison: No Pre-Cert rows at all.

Tia annotated each aggregate section with "Missing on comparison."

**Likely cause:** Pre-Cert Lists are structurally different from benefit rows — they are
long lists of service names rather than benefit-parameter value pairs. The model may be
skipping them because they don't fit the expected comparison table schema.

**Expected behavior:** Pre-Cert List items should be compared and differences surfaced.
At minimum, the presence/absence of items on each list should be flagged.

### LE-08: Missing Out-of-Network Row for In-Network/Out-of-Network Pairs

| | |
|---|---|
| **Severity** | Medium |
| **Frequency** | Observed once (RCP2_HYD1, Deductible section) |
| **Test doc** | RCP2_HYD1 |

**Problem:** When a benefit has both In-Network and Out-of-Network rows, the comparison
sometimes captures only the INN row and omits the OON row.

**Evidence:** Deductible section — comparison rows 70–74 show:
- Row 70: Deductibles – Individual, INN → Mismatch
- Row 71: Deductibles – Individual, OON → Mismatch
- Row 72: Deductibles – Family, INN → Mismatch
- Row 73: Deductibles – Family, OON → Mismatch
- Row 74: Deductible Standardly Applied – Non Routine Services, **INN only** → Mismatch

The source document clearly has both INN and OON for the "Deductible Standardly Applied"

**Relationship to LE-07:** This is a specific case of incomplete row coverage, but notable
because INN/OON pairs should always come in pairs. Other deductible items correctly show
both.

**Expected behavior:** If a benefit has In-Network and Out-of-Network settings, both must
appear in the comparison.

### LE-09: False Match on Benefit Details (Match/Mismatch Logic Errors)

| | |
|---|---|
| **Severity** | High |
| **Frequency** | Observed across two benefits in deviation-based comparison (PTG1) |
| **Test doc** | PTG1 (deviation template) |

**Problem:** The comparison marks benefits as "Match" when the underlying detail values
clearly differ between the template and aggregate. Two sub-cases:

**a) Nutritional Counseling (PTG1):**
- The template (FOB Standard) shows "Covered, 80%, Ded Yes, Copay N/A" for INN settings
  and "Covered, 60%, Ded Yes, Copay N/A" for OON settings. Expected Value is "Not
  Covered" (this is a deviation document, so the expected column reflects what should
  differ from standard).
- The aggregate returns "Covered – Conditionally, 80%, Ded Yes, Copay N/A" (note:
  "Conditionally" qualifier added) and "Covered – Conditionally, 50%, Ded Yes, Copay N/A"
  (note: 50% vs 60% difference).
- The comparison marks all Nutritional Counseling rows as **"Match"** with details starting
  with "Benefit su..." (truncated).
- The source documents show significant differences: the aggregate adds limits (30 Visits
  Per Year), conditional coverage language, and an "All Other: Covered: No" sub-provision
  that the standard template doesn't have.

**b) Home Visits:**
- The template includes detailed limits: "Limited to 90 days per benefit period. 1 Visit =
  up to 4 hours. Patient is required to be homebound. Visits do count towards the Home
  Health visit limit."
- The aggregate shows basic coverage parameters (80% Ded Yes, Copay N/A) but **no limits
  or notes**.
- The comparison marks Home Visits as **"Match"** — ignoring the missing limit information
  entirely.

**Expected behavior:** When the coverage parameters (percentages, limits, conditions,
notes) differ, the status must be Mismatch. The comparison should evaluate the full cell
content, not just the benefit name and top-level coverage indicator.

## Cross-Cutting Themes

The nine issues cluster into three root-cause categories:

### 1. Completeness — Rows/fields being dropped (LE-01, LE-06, LE-07, LE-08)

The model is not exhaustively extracting every row from the source documents. This
manifests as:
- Entire sections omitted (Pre-Cert Lists — LE-06)
- Tail-end items within sections dropped (LE-07)
- One side of INN/OON pairs missing (LE-08)
- Aggregate-only fields invisible (LE-01)

These likely share a common root: the model's output generation is either hitting token
limits, prioritizing "important" rows, or following the template structure as the sole
enumeration guide rather than treating both documents symmetrically.

### 2. Comparison Accuracy — Match/mismatch logic too loose (LE-04, LE-05, LE-09)

The model is applying semantic understanding where literal comparison is required:
- Treating "12 Months" = "365 Days" as equivalent (LE-05)
- Matching on benefit name while ignoring detail differences (LE-09)
- Ignoring qualifier text in merged cells (LE-04)

These are prompt engineering and comparison logic issues. The model needs explicit
instruction that FOB audits require character-level comparison of all text, and that
semantic equivalence should be noted but still flagged as Mismatch.

### 3. Consistency — Non-deterministic output (LE-02, LE-03)

The same inputs produce different levels of granularity across runs:
- 125-row swing between identical runs (LE-02)
- Benefits correctly broken out in one analysis but concatenated in another (LE-03)

This is an inherent LLM challenge but can be mitigated through more prescriptive prompts,
structured output schemas, temperature controls, and/or validation passes.
